# 功能規格：B2C 預約系統後端

**功能分支**: `001-b2c-booking-backend`
**建立日期**: 2026-01-17
**狀態**: 草稿
**輸入**: 使用者描述: "建立一個通用的 B2C 預約系統後端規格..."

## 使用者場景與測試 *(必要)*

### 使用者故事 1 - Provider 管理可用時間 (優先級: P1)

作為一個 **Provider**，我希望能設定我在特定日期的可用時間區段，並管理我的預約請求，以便我能完全控制我的排程並有效接受客戶預約。

**為何此優先級**: 這是系統的核心功能，Provider 無法設定時間，客戶就無法預約。

**獨立測試**: Provider 可以成功建立、查看、刪除特定日期的可用時段。當預約進來時，Provider 能看到請求並進行確認或取消。

**驗收場景**:

1.  **假設** Provider 登入系統，**當** Provider 為特定日期 (例如 2026-10-20) 提交一個或多個可用時間段 (例如 09:00-11:00, 14:00-16:00)，**那麼** 系統應儲存這些時段，並在查詢時返回。
2.  **假設** Provider 有一個待處理的預約請求，**當** Provider 確認該預約，**那麼** 該預約狀態應更新為 "Confirmed"，並且對應的可用時段應標記為已預訂。
3.  **假設** Provider 建立的可用時段有重疊 (例如 09:00-11:00 和 10:00-12:00)，**當** 提交時，**那麼** 系統應返回錯誤，防止建立重疊的時段。

---

### 使用者故事 2 - Customer 預約服務 (優先級: P1)

作為一個 **Customer**，我希望能瀏覽特定 Provider 的可用服務與時間，並能輕鬆建立預約，以便我能快速方便地預訂我需要的服務。

**為何此優先級**: 這是系統的核心商業流程，讓消費者能夠實際使用系統進行預約。

**獨立測試**: Customer 可以查詢到特定 Provider 在某個日期範圍內的可用時段，並成功提交一個預約申請。

**驗收場景**:

1.  **假設** Provider 在 2026-10-20 有一個可用時段 14:00-15:00，**當** Customer 查詢該 Provider 在該日期的可用時段，**那麼** 系統應返回 14:00-15:00 這個時段。
2.  **假設** 一個可用時段存在，**當** Customer 針對該時段提交預約請求，**那麼** 系統應建立一個狀態為 "Pending" 的 Booking 紀錄，並通知 Provider。
3.  **假設** 一個時段已被其他 Customer 確認預約，**當** 另一個 Customer 嘗試預約同個時段，**那麼** 系統應返回錯誤，告知該時段不可用。

---

### 使用者故事 3 - Admin 監控系統 (優先級: P2)

作為一個 **Admin**，我希望能查看系統中所有的使用者和預約紀錄，以便我能進行系統管理、數據分析和問題排查。

**為何此優先級**: 管理功能對於系統的長期運營和維護至關重要，但優先級略低於核心交易流程。

**獨立測試**: Admin 登入後，可以成功查詢到系統中的使用者列表和所有預約紀錄列表。

**驗收場景**:

1.  **假設** 系統中有多個使用者和預約，**當** Admin 訪問使用者管理頁面，**那麼** 應能看到所有使用者列表。
2.  **假設** 系統中有多個預約，**當** Admin 訪問預約管理頁面，**那麼** 應能看到所有預約紀錄，並能依狀態、日期等條件進行篩選。

## 需求 *(必要)*

### 功能需求

- **FR-001**: 系統 **必須** 支援 Provider, Customer, Admin 三種角色。
- **FR-002**: Provider **必須** 能夠為特定日期設定一個或多個非連續的可用時間區段 (AvailabilitySlot)。
- **FR-003**: Provider **必須** 能夠批次建立可用時段。
- **FR-004**: Customer **必須** 能夠查詢特定 Provider 在指定日期範圍內的可用時段。
- **FR-005**: Customer 建立預約時，系統 **必須** 檢查時間衝突，防止同一時段被重複預約 (原子性操作)。
- **FR-006**: Booking 紀錄 **必須** 包含 `Pending`, `Confirmed`, `Completed`, `Cancelled` 四種狀態。
- **FR-007**: 一個 AvailabilitySlot 在被 `Confirmed` 的 Booking 關聯後，**必須** 變為不可預約狀態。
- **FR-008**: Service, AvailabilitySlot, Booking 的數據模型 **必須** 包含預留擴充欄位 (例如 JSON/TEXT 類型)，以支援未來的「地點」或「線上會議連結」。
- **FR-009**: Admin **必須** 能夠查看系統中所有的使用者和預約紀錄。

### 關鍵實體

- **Service (服務)**: 代表提供者可提供的服務項目。
  - `Name` (名稱): String
  - `Description` (描述): Text
  - `DurationInMinutes` (時長): Integer
  - `Price` (價格): Decimal
- **AvailabilitySlot (可用時段)**: Provider 設定的、可被預約的時間區塊。
  - `ProviderId`: FK to User
  - `StartTime`: DateTime
  - `EndTime`: DateTime
  - `IsBooked`: Boolean (或透過 Booking 存在與否來判斷)
- **Booking (預約紀錄)**: Customer 發起的預約。
  - `CustomerId`: FK to User
  - `ProviderId`: FK to User
  - `ServiceId`: FK to Service
  - `SlotId` (或直接儲存時間): FK to AvailabilitySlot
  - `Status`: Enum (Pending, Confirmed, Completed, Cancelled)
  - `ExtendedProperties`: JSON/Text (用於儲存地點、會議連結等未來資訊)

## 成功標準 *(必要)*

### 可衡量成果

- **SC-001**: Provider 能在 60 秒內完成為一整天的非連續時段排班。
- **SC-002**: Customer 查詢可用時段的 API 回應時間，在 95% 的情況下應低於 500 毫秒。
- **SC-003**: 預約建立的成功率需達到 99.9%，並能完全防止超額預約。
- **SC-004**: 新的 Provider 能在 5 分鐘內完成個人檔案和第一個可用時段的設定，無需額外教學。
